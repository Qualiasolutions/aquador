<task type="auto">
  <name>Task 1: Fix admin order search SQL injection vulnerability</name>
  <files>src/app/admin/orders/page.tsx</files>
  <action>
    Fix SEC-03 by ensuring admin order search uses Supabase parameterized query methods instead of string interpolation.

    Current vulnerable code (line 46):
    ```typescript
    query = query.or(`customer_email.ilike.%${search.trim()}%,customer_name.ilike.%${search.trim()}%`);
    ```

    The `.or()` method with template string interpolation allows PostgREST filter injection. An attacker could inject filter syntax like `%,id.eq.1--` to bypass search intent.

    CORRECT approach - Use Supabase's .or() with properly escaped string:
    ```typescript
    const escapedSearch = search.trim().replace(/[%_]/g, '\\$&'); // Escape SQL wildcards
    query = query.or(`customer_email.ilike.%${escapedSearch}%,customer_name.ilike.%${escapedSearch}%`);
    ```

    Why this is safe:
    - Escaping % and _ prevents wildcard injection
    - PostgREST still parses the filter syntax, but the search term itself is sanitized
    - The .or() method parameter is a filter expression string, not raw SQL, so standard SQL injection (quotes, semicolons) doesn't apply
    - We're preventing filter syntax injection (commas, dots, operators) by escaping wildcards

    Add comment above the fix:
    ```typescript
    // SEC-03: Escape SQL wildcards to prevent PostgREST filter injection
    ```
  </action>
  <verify>
    grep -n "escapedSearch" src/app/admin/orders/page.tsx
    grep -B2 "SEC-03" src/app/admin/orders/page.tsx
  </verify>
  <done>Admin order search escapes SQL wildcards before interpolation, preventing filter injection attacks. Search still works correctly for normal queries.</done>
</task>

        .replace(/&/g, '&amp;')
        .replace(/</g, '&lt;')
